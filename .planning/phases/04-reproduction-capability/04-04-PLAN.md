**Module docstring:** `"""Core reproduction engine — orchestrates issue reproduction from diagnosis.\n\nThe ReproductionEngine runs a prepare → execute → report workflow,\ncalling exec_shell for command execution (local or remote via SSH),\nand produces a structured ReproductionResult for Phase 5 consumption.\n"""`

Module-level logger: `logger = logging.getLogger(__name__)`

**Class: `ReproductionEngine`:**

1. **Constructor `__init__(self, router: ModelRouter, repo_path: str, settings: Settings | None = None)`:**
   - Store `self._router = router`
   - Store `self._repo_path = Path(repo_path).resolve()` (resolved once, per D-10)
   - Store `self._settings = settings or Settings()` (default fallback)

2. **Public method `async def reproduce(self, diagnosis: DiagnosisResult) -> ReproductionResult`:**
   - Step 1 (prepare): Call `self._detect_venv()` to get env vars (D-14). Merge with `self._settings.env_vars`.
   - Step 2 (execute): For each hypothesis in `diagnosis.hypotheses` (up to 3):
     a. Call `self._build_reproduction_command(hypothesis)` to construct the command string.
     b. Validate any file paths in the command via `self._validate_path(path)` (D-10).
     c. Import exec_shell locally: `from ascend_agent.tools.shell_exec import exec_shell`
     d. Record start time: `start = time.monotonic()`
     e. Call `await exec_shell(command, timeout=self._settings.shell_timeout)` (D-07, timeout from settings)
     f. Parse result: `result = json.loads(result_json)`
     g. Calculate duration: `duration = time.monotonic() - start`
     h. Build ReproductionResult with status, command, stdout, stderr, exit_code from parsed result, duration_seconds=duration, hypothesis_id_tested=i (0-based index, D-12), files_changed=[] (empty for now)
   - Step 3 (report): Return the first ReproductionResult (simplest case) or aggregate if multiple hypotheses tested.
   - If no hypotheses: return ReproductionResult with status="error", command="", stderr="No hypotheses to test", exit_code=-1, duration_seconds=0, hypothesis_id_tested=-1, files_changed=[].
   - If exec_shell raises an exception: catch it, log via `logger.error("Reproduction failed: %s", exc)`, return ReproductionResult with status="error", stderr=str(exc), exit_code=-1.

3. **Private method `_detect_venv(self) -> dict[str, str]`:**
   - Check `os.environ`: if "VIRTUAL_ENV" is set, add to returned dict
   - Check `os.environ`: if "CONDA_PREFIX" is set, add to returned dict
   - Return the dict (may be empty) — D-14: engine detects but does NOT create/manage venvs

4. **Private method `_build_reproduction_command(self, hypothesis) -> str`:**
   - Per research resolution: use heuristic command construction (not LLM). Start simple: run the Python file from evidence.
   - If hypothesis has evidence with a file_path ending in `.py`: return `f"python {evidence_file}"`
   - If hypothesis has evidence referencing a test: return `f"python -m pytest {evidence_file}"`
   - Fallback: `f"echo 'No actionable command for: {hypothesis.root_cause[:80]}'"`
   - Command strings use simple f-strings with evidence file paths — no shell injection concerns since file paths come from DiagnosisResult (trusted internal data, not user input)

5. **Private method `_validate_path(self, path_str: str) -> bool`:**
   - Per D-10: reuse the exact path traversal pattern from edit_file (lines 50-59 of file_edit.py)
   - `path = Path(path_str).resolve()`
   - `return str(path).startswith(str(self._repo_path))`
   - Paths within repo → True, paths outside → False

6. **Private method `_check_dependencies(self) -> list[str]`:**
   - Per D-13 and research resolution: for LOCAL execution, check that Python and pip are available; for REMOTE, check only.
   - For Phase 4 scope: check that `python --version` runs successfully as a basic sanity check.
   - Returns list of missing dependency names (empty = all present).
   - Calls exec_shell with `"python --version"` to verify Python is available.
   - Per research Open Question 2 resolution: do NOT auto-install packages remotely. Local: check + report. Remote: check + report (not auto-install).

**Logging:** Use `logger.info()` for workflow steps (e.g., "Reproducing hypothesis 0/2"), `logger.error()` for failures. Follow the Engine pattern from diagnosis/engine.py lines 186-188.

**Error handling pattern:** Follow diagnosis/engine.py lines 262-283 — catch Exception, log error, return a valid ReproductionResult with status="error".

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ReproductionEngine.reproduce() → exec_shell | Command string crosses into shell execution tool |
| evidence file_path → _build_reproduction_command | File paths from diagnosis data used to construct commands |
| user file paths → _validate_path | Untrusted paths checked against repo boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | _build_reproduction_command shell injection | accept (low risk) | Command strings are built from diagnosis evidence (trusted internal data — Hypothesis.evidence[].file_path), not from user input. File paths originate from the diagnosis engine, which reads them from the codebase. No `shlex.quote()` is applied to evidence file paths because they are not user-supplied. If future phases add user-supplied command parameters, `shlex.quote()` must be added. |
| T-04-02 | Elevation of Privilege | _validate_path path traversal | mitigate | Exact pattern from edit_file (Phase 3 D-15): `Path(path_str).resolve()` then `str(resolved).startswith(str(self._repo_path))`. Any path resolving outside the repo boundary returns False and is rejected before command construction. This blocks `../` and symlink attacks. |
| T-04-06 | Tampering | _build_reproduction_command env var injection | mitigate | Command construction uses f-strings with evidence file paths only — no environment variable interpolation. The env var dict from _detect_venv is not interpolated into command strings; it is passed separately to exec_shell (or would be if subprocess env were overridden). |

**T-04-01 disposition: accept (low risk).** Command strings in Phase 4 are constructed from trusted internal data:
- Source: `Hypothesis.evidence[].file_path` — populated by the diagnosis engine from AST-parsed codebase files
- No user input flows into _build_reproduction_command
- The heuristic is `"python {evidence_file}"` where evidence_file is a file path confirmed to exist in the repo
- Risk of injection via file name: mitigated by _validate_path (file must be within repo) and the fact that file names in a codebase are controlled by developers, not attackers

If Phase 5 or future phases add user-supplied reproduction command parameters, `shlex.quote()` must be applied at that point.

**T-04-02 disposition: mitigate.** Path traversal uses the same proven pattern as edit_file (Phase 3 D-15, ASVS V4). Implementation: `Path(path_str).resolve()` normalizes the path (resolves ../ and symlinks), then checks `str(resolved).startswith(str(self._repo_path))`. Paths resolving outside the repo are rejected before any command is constructed.
</threat_model>
