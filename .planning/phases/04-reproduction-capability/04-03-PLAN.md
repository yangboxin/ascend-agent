<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client → exec_shell | Untrusted command string crosses into shell execution |
| exec_shell → local subprocess | Command string enters OS shell — shell injection risk |
| exec_shell → remote SSH | Command traverses network to remote host — MITM + auth risk |
| env vars → exec_shell | ASCEND_SSH_* env vars control execution path |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | exec_shell command parameter | accept (delegated) | exec_shell passes command strings through without sanitization. Shell injection prevention is delegated to the caller (ReproductionEngine). The tool itself runs whatever command it receives. This is a documented architectural decision. |
| T-04-03 | Spoofing | _exec_remote SSH connection | mitigate | `known_hosts=None` disables host key verification — acceptable for internal test machines behind firewall. Human checkpoint (Plans 04-05 Task 3) verifies environment awareness. If production use is needed, user must configure `known_hosts` to a trusted file. |
| T-04-04 | Denial of Service | _exec_local, _exec_remote timeout | mitigate | `asyncio.wait_for(proc.communicate(), timeout)` and `asyncio.wait_for(conn.run(), timeout)` enforce configurable timeout. Default 60s from `shell_timeout` setting. Timeout triggers process kill (local) or raises TimeoutError (remote), both captured as status="error". |
| T-04-05 | Information Disclosure | SSH key material in _exec_remote | mitigate | SSH key path stored in env var ASCEND_SSH_KEY_PATH. asyncssh `client_keys` parameter reads key file directly — key material never passed via command line, never logged. Conditional import of asyncssh prevents key module from being loaded when SSH path not used. |

**T-04-01 disposition: accept (delegated).** exec_shell is a general-purpose command execution tool. It does not parse, validate, or sanitize command strings because:
1. The ReproductionEngine constructs commands from diagnosis evidence (file paths, test names) — not arbitrary user input
2. Shell injection prevention via `shlex.quote()` is applied at the Engine layer where the command is constructed, not at the execution layer
3. Adding sanitization here would limit legitimate use cases (e.g., piped commands, redirects)
This is documented in the tool's implementation comments and the engine's threat model.

**T-04-03 disposition: mitigate.** `known_hosts=None` is set explicitly for automated test machine access. This trades host verification for operational simplicity. For production environments, users can override by setting up `~/.ssh/known_hosts` (asyncssh reads it by default when `known_hosts` is not set to `None`).
</threat_model>
