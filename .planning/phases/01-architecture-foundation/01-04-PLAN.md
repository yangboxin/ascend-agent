Create `src/ascend_agent/tools/code_search.py` with the code search tool function.

Function signature:
```python
async def search_code(pattern: str, path: str = ".", ctx: Context) -> str:
    """Search for a regex pattern in Python files within the specified path.
    
    Args:
        pattern: Regex pattern to search for
        path: Root directory to search in (defaults to ".")
    """
```

Implementation:
1. Try `subprocess.run(["rg", "-n", pattern, path, "--type", "py", "--no-heading"], capture_output=True, text=True, timeout=30)` first (ripgrep path).
2. On `FileNotFoundError` (rg not installed) or any non-zero returncode: fall back to Python-native search.
3. Python-native fallback:
   - Use `os.walk(path, topdown=True)` — filter dirs: skip `.`-prefixed dirs, `__pycache__`, `node_modules`
   - For each `.py` file, open with `encoding="utf-8", errors="replace"`
   - For each line, use `re.search(pattern, line)` — if match, collect `relpath:lineno:line.rstrip()[:200]`
   - Truncate results to 500 matches
   - Return `"\n".join(results)` or `f"No matches found for '{pattern}'"` if empty
4. Log progress via `await ctx.info(f"Searching for '{pattern}' in {path}")` — never `print()`.
5. Wrap in try/except:
   - `subprocess.TimeoutExpired` → `raise ValueError(f"Search timed out for pattern '{pattern}'")`
   - General `Exception` → `raise ValueError(f"Search failed: {e}")`
6. Truncate return string to 10000 characters to avoid MCP message size limits. If truncated, append `\n... (truncated at 10000 chars)`.

Use `from mcp.server.fastmcp import Context`.
Use `import re, os, subprocess, pathlib`.
</action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP server STDIO ↔ client | Inter-process communication via stdin/stdout |
| search_code(pattern, path) → filesystem | User-supplied regex pattern and path read from disk |
| search_code(pattern, path) → subprocess | Optional ripgrep subprocess execution |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-08 | Tampering | server.py / all tool functions | mitigate | All MCP tool inputs validated through Pydantic models per D-18. FastMCP uses JSON Schema generated from function type annotations to validate inputs before the function is called. |
| T-01-09 | Tampering | code_search.py (STDIO) | mitigate | NEVER use `print()` — stdout is the MCP transport channel. All logging uses `ctx.info()` or `stderr`. Enforced via grep gate: `grep -n "^[^#]*\bprint\b" src/ascend_agent/tools/` must return empty. |
| T-01-10 | Tampering | code_search.py (subprocess) | mitigate | `subprocess.run()` with fixed args `["rg", "-n", pattern, path, "--type", "py"]` — user-supplied `pattern` and `path` could inject shell options. Mitigation: use `subprocess.run()` with list args (no `shell=True`), preventing shell injection. The `pattern` is passed as a single arg to rg. Timeout of 30s prevents resource exhaustion. |
| T-01-11 | Tampering | code_search.py (path traversal) | accept | The `path` parameter controls file system scope. In Phase 1, the MCP server trusts its caller (Phase 2 orchestrator). Path traversal would be a risk if the MCP server were exposed over network, but STDIO transport limits access to the local process that launched it. Accept for Phase 1. |
| T-01-12 | Denial of Service | code_search.py (ReDoS) | mitigate | User-supplied `pattern` could cause ReDoS via pathological regex. Mitigation: wrap `re.search()` in `re._compile()`? No — practical mitigation: use `re.search(pattern, line, timeout=1)` (Python 3.11+) or add a custom timeout wrapper. For Python 3.10 compatibility (current environment), accept this risk for Phase 1. Document that Phase 2 should add `regex` library with timeout. Mitigation for now: truncate lines to 200 chars before regex matching, limiting backtracking. |
</threat_model>
