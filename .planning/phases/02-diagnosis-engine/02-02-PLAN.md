<objective>
Implement the core diagnosis engine: the Engine class with LLM-driven search loop and AST-based function body extraction utility. The Engine orchestrates up to 3 LLM search iterations, calls the code_search MCP tool for source code lookups, reads function bodies at ±5 lines scope, and produces ranked hypotheses with evidence.

Purpose: Deliver the central orchestration layer (DIAG-01, DIAG-02) — an LLM reviews parsed trace frames, decides what to search for (D-10), receives code search results, and either searches again or produces a diagnosis. The engine is silent (D-03), reports partial results on failure (D-04), and does not follow cross-references (D-12).
Output: `src/ascend_agent/diagnosis/engine.py` with Engine class + `_read_function_body()` utility; `tests/test_diagnosis/test_engine.py` with search loop, partial failure, and code extraction tests; conftest.py updated.

Create `src/ascend_agent/diagnosis/engine.py` with two components:

### 1. `_read_function_body(file_path: str, target_line: int, context_lines: int = 5) -> str | None`

Per D-09/D-11: AST-based function boundary detection with line-based fallback.

Implementation steps:
- Read file, parse with `ast.parse(source)`. Use `compile()` mode as `exec`.
- Walk AST nodes looking for `ast.FunctionDef` and `ast.AsyncFunctionDef`.
- For each function node, check if `node.lineno <= target_line <= node.end_lineno`.
- If found: extract source lines from `max(0, node.lineno - 1 - context_lines)` to `min(len(lines), node.end_lineno + context_lines)`. Return as newline-joined string with `"{line_no}:{line}"` prefix per line (matching the code_search output format).
- If NOT found (fallback): extract lines from `max(0, target_line - 1 - context_lines)` to `min(len(lines), target_line + context_lines)`. Same formatting.
- If file doesn't exist or any error occurs: return None (caller handles logging).
- Import ast at module top. Handle `SyntaxError` in ast.parse — fallback to line window in that case.

### 2. `Engine` class

**Constructor:** `Engine(router: ModelRouter, repo_path: str)`
- Store `self._router` and `self._repo_path` as private attributes.
- Store `self._repo_path_resolved = pathlib.Path(repo_path).resolve()`.

**`diagnose(self, context_doc: "ContextDocument") -> DiagnosisResult`** — core method:

1. Build initial messages list:
   - System prompt (string): instructs LLM it is a Python debugging expert. Must include: "Do not ask clarifying questions. You have up to 3 searches to gather code context. After each search result, decide: search again with a more specific pattern, or produce your top 3 hypotheses ranked by confidence." (D-03, D-02, D-06)
   - User message: formatted trace info. Include the error type, error message, and each frame's file:line:function. Include the repo path so the LLM can assess which frames are in-repo.

2. Loop for `iteration` in `range(1, MAX_ITERATIONS + 1)` where MAX_ITERATIONS = 3:
   - Call `self._router.completion(messages=messages + search_history_formatted, response_model=SearchDecision, max_tokens=4096)`.
   - If `decision.action == "hypothesize"`: call `self._generate_hypotheses(messages + search_history)` and return.
   - For each search in `decision.searches[:3]`: call `self._execute_search(search.pattern)` and append to search_history.

3. If loop completes (budget exhausted): call `self._generate_hypotheses(messages + search_history, exhausted=True)` and return.

**`_generate_hypotheses(messages, exhausted=False) -> DiagnosisResult`**
- Add appropriate user message: if exhausted, "Search budget exhausted. Generate your best hypotheses with available information." otherwise "Generate your top 3 ranked hypotheses with evidence."
- Call `self._router.completion(messages=messages + final_prompt, response_model=DiagnosisResult, max_tokens=8192)` — per Pitfall 1, hypothesis generation may need more tokens.
- Set `result.iterations_used` to the number of iterations used.

**`_execute_search(self, pattern: str) -> dict`**
- Import `search_code` from `ascend_agent.tools.code_search` (local import to avoid circular dependency).
- Use `asyncio.run(search_code(pattern, resolved_repo_path))`.
- On success: try to read function bodies for matched lines using `_read_function_body`. Append code snippets to the result.
- On any exception: return dict with `{"error": str(error)}`. This gets formatted into search history.

**`_build_system_prompt()`** and **`_build_user_prompt(context_doc)`** as static/helper methods to construct the initial messages. These are functions at module level, not methods on Engine.

**`_format_search_history(search_history: list)`** — helper: formats the accumulated search results into a readable text block for the LLM. Each entry shows `Search {N}: pattern="{pattern}"`, followed by results. Failed searches show error messages.

### Design constraints enforced in code:
- No clarifying questions: system prompt asserts "Do not ask clarifying questions." Engine never prompts user for input (D-03).
- No cross-reference following: Engine only searches patterns determined by LLM from trace frames. No automatic import/callee resolution (D-12).
- Partial results on failure: `_execute_search` catches exceptions and returns error dict, never raises. Engine still produces DiagnosisResult even when searches fail (D-04).
- Max 3 iterations: enforced by loop bound. LLM cannot request more searches (D-02).

| T-2-02 | Spoofing | Engine.search loop | mitigate | LLM output constrained by SearchDecision response_format Pydantic schema (pattern= r'^(search\|hypothesize)$'). System prompt asserts authority: "You are a Python debugging expert. Analyze the following stack trace..." |
| T-2-03 | Denial of Service | _execute_search | mitigate | code_search tool has built-in 30s timeout. Engine catches all exceptions from search_code and converts to PartialFailure — never crashes. |

</threat_model>
